# Overview

This specification completed the Retrieval Observability layer for the CRM Copilot RAG pipeline.

Following the migration of Spec 005 to LangChain, the observability layer was updated to integrate seamlessly with the new retrieval architecture while preserving the original repository, service, and API design.

The goal of this specification is to record every semantic retrieval performed by the RAG pipeline, allowing retrieval quality, latency, and execution behavior to be inspected for debugging, monitoring, evaluation, and future analytics.

Every retrieval request can now be traced from the original user query through the retrieved LangChain Documents together with similarity scores, retrieval metadata, and performance metrics.

# LangChain Architecture Integration

Spec 005 introduced a LangChain-based RAG foundation.

Spec 006 was updated accordingly.

Current retrieval pipeline:

```
User Query
      │
      ▼
RetrievalService
      │
      ▼
Retriever
      │
      ▼
PGVectorStore
      │
      ▼
DocumentChunkRepository
      │
      ▼
PostgreSQL + pgvector
```

The retrieval layer now returns LangChain `Document` objects together with similarity scores while automatically recording observability information.

# Database Models

## RetrievalTrace

Stores one record for every retrieval execution.

Captured fields include:

- Trace ID
- Tenant ID
- Conversation ID
- Query
- Embedding model
- Vector store
- Embedding latency
- Retrieval latency
- Total latency
- Retrieved chunk count
- Retrieval metadata
- Retrieval status
- Error message
- Created timestamp
- Updated timestamp

Relationships:

- Conversation
- RetrievedChunk

---

## RetrievedChunk

Stores every chunk returned during retrieval.

Captured fields include:

- Retrieval Trace ID
- Document ID
- Document Chunk ID
- Rank
- Similarity score
- Chunk preview

Relationships:

- RetrievalTrace
- KnowledgeDocument
- DocumentChunk

# LangChain Retrieval Metadata

RetrievalService now records retrieval metadata generated by the LangChain Retriever.

Current metadata includes:

- Retriever configuration
    - top_k
    - score_threshold
- Applied filters
    - document_id

This metadata is persisted with every retrieval trace and provides additional context for debugging retrieval behavior.

The metadata model is intentionally extensible to support future additions such as:

- Retriever strategy
- Embedding provider
- Vector store implementation
- Reranker configuration
- Search type
- Additional retrieval parameters

# Retrieval Layer Integration

Implemented a dedicated RetrievalService responsible for:

- Executing semantic retrieval
- Measuring retrieval latency
- Recording retrieval traces
- Logging retrieved chunks
- Persisting retrieval metadata
- Recording retrieval failures
- Returning RetrievalResult objects

Responsibilities intentionally exclude:

- Prompt construction
- Context assembly
- LLM generation

These responsibilities belong to the RAGChain introduced in Spec 005.
